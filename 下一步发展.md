# 下一步发展

## 目标

后续版本重点从“本地功能模块”升级为“带授权验证、WebUI 管理、服务端卡密控制”的完整模块体系。

核心目标：

- 防止模块被随意传播和多人共用。
- 保留现有设备伪装、CPU 伪装、白名单、越狱隐藏环境、冻结多开系统软件功能。
- 增加授权验证，但不能影响 Zygisk 正常加载。
- 认证失败时只关闭伪装功能，不让模块崩溃或断开 Zygisk。
- 增加 KernelSU WebUI，方便用户操作。

## 一、授权验证方案

### 1. 设备唯一 ID

使用 root 权限从 `/proc/cmdline` 或 `/proc/bootconfig` 中读取硬件级启动参数。

优先级如下：

```text
1. androidboot.cpuid        小米/红米/黑鲨
2. androidboot.chipid       OPPO/一加/真我
3. androidboot.emmcid       红魔/努比亚
4. oplusboot.serialno       OPPO 备选
5. androidboot.serialno     通用兜底
```

模块不按品牌硬判断，而是在 C++ 内顺序尝试读取，取第一个非空值。

最终生成：

```text
device_hash = SHA256(hardware_id + compile_time_salt)
```

原始硬件 ID 不直接上传、不明文保存，只上传哈希。

### 2. 卡密绑定逻辑

卡密不永久死绑设备，但同一卡密同一时间只允许一个设备通过验证。

逻辑：

```text
设备 A 使用卡密 → 验证通过 → A 成为当前活跃设备
设备 B 使用同一卡密 → 验证通过 → B 成为当前活跃设备，A 被互抵
设备 A 下次验证 → 返回异地登录信息 → 关闭所有伪装功能
```

这样可以允许用户换机，但不允许多人同时共用同一卡密。

### 3. 本地授权文件

服务端返回签名后的 `license.json`。

示例：

```json
{
  "code_id": "GWBHJ-XXXX-XXXX-XXXX",
  "device_hash": "sha256...",
  "active": true,
  "revoked": false,
  "last_server_ok": 1719012345,
  "check_interval": 600,
  "grace_seconds": 86400,
  "lock_until": null,
  "clash_count": 0,
  "signature": "ed25519_signature"
}
```

### 4. 二进制内验签

授权判断必须放在 `.so` 中，不依赖 shell 明文逻辑。

`.so` 内置：

- 服务端公钥
- 设备 ID 生成逻辑
- license 验签逻辑
- 授权状态判断逻辑

私钥只放服务端，绝不进入模块。

认证函数：

```cpp
static bool authOk() {
    // 读取 license.json
    // Ed25519 验签
    // 校验 device_hash 是否匹配本机
    // 校验 revoked 是否为 false
    // 校验 lock_until 是否过期
    // 校验 last_server_ok + grace_seconds 是否有效
    // 全部通过才返回 true
}
```

Zygisk 逻辑：

```cpp
void preAppSpecialize(...) {
    if (!authOk()) {
        api->setOption(zygisk::Option::DLCLOSE_MODULE_LIBRARY);
        return;
    }

    // 授权通过后才执行白名单、设备伪装、CPU 伪装
}
```

认证失败时：

- 不修改 `Build.*`
- 不修改 `ro.product.*`
- 不 bind mount `/proc/cpuinfo`
- 不影响 Zygisk 正常加载

## 二、服务端方案

### 1. 初期部署

使用腾讯 EdgeOne Edge Functions。

原因：

- 国内访问速度好
- 免费套餐可用
- 自带 HTTPS
- 方便后期迁移

### 2. 后期迁移

用户量增大后迁移到独立服务器。

迁移成本低：

- API 路径保持不变
- 客户端只需修改 `SERVER_URL` 后重新编译
- Ed25519 密钥对可继续沿用
- KV 数据可导出为 JSON 后导入数据库

服务器 1T/月流量足够使用。

### 3. API 设计

```text
POST /api/bind
首次绑定卡密

POST /api/verify
定期验证授权

POST /api/clash-info
查询互抵详情

POST /admin/generate
批量生成卡密

POST /admin/delete
删除卡密

GET /admin/list
查看卡密列表
```

### 4. 数据结构

```json
{
  "code": "GWBHJ-XXXX-XXXX-XXXX",
  "status": "unused | active | locked | permanent_locked",
  "active_device_hash": "sha256...",
  "active_device_info": {
    "brand": "HUAWEI",
    "model": "SGT-AL10",
    "serial": "HW1234567890"
  },
  "active_time": "2026-06-22T10:00:00Z",
  "clash_count": 0,
  "lock_until": null,
  "clash_log": []
}
```

## 三、互抵和锁死机制

### 1. 互抵提示

当设备被其他设备使用同一卡密顶掉时，WebUI 显示：

```text
❌ 卡密异地登录，请妥善管理卡密
⚠️ 连续互抵将导致卡密锁死
连续互抵次数: 2/3 (锁死: 1小时)
[查看互抵详情]
```

### 2. 互抵详情

详情中显示：

- 对方设备型号
- 对方设备序列号
- 登录时间
- 设备 hash 摘要
- 当前设备是否被踢

示例：

```text
2026-06-22 10:00
设备: OPPO Find X7
序列号: OP9876543210
操作: 踢掉了你的设备
```

### 3. 锁死规则

```text
互抵 1-3 次：锁 1 小时
互抵 4-5 次：锁 4 小时
互抵 6-8 次：锁 1 天
互抵 9 次：锁 1 周
互抵 10 次：永久锁死
```

锁死期间：

- 不允许绑定
- 不允许验证通过
- WebUI 显示剩余锁定时间
- 显示继续互抵的后果

## 四、WebUI 方案

KernelSU 支持 WebView，可以添加模块 WebUI。

目录：

```text
module/webroot/
├── index.html
├── style.css
└── script.js
```

### 1. WebUI 功能

- 查看授权状态
- 输入卡密绑定
- 查看互抵/锁死状态
- 查看设备 ID
- 查看当前序列号
- 添加/删除白名单包名
- 查看模块运行状态
- 执行 action 功能
  - 修改序列号
  - 越狱隐藏环境
  - 冻结多开系统软件

### 2. WebUI 调用 action.sh

`action.sh` 增加命令行参数模式：

```sh
sh action.sh reset-serial
sh action.sh jailbreak
sh action.sh freeze
sh action.sh bind GWBHJ-XXXX-XXXX
sh action.sh status
```

无参数时保留音量键交互菜单。

## 六、安全风险与应对

### 1. 可防御风险

| 风险 | 应对 |
|------|------|
| 修改 license.json | Ed25519 验签失败 |
| 伪造服务器响应 | 无私钥无法签名 |
| 修改 auth.status | `.so` 不读取此文件 |
| 多人共用卡密 | 互抵 + 渐进锁死 |
| 抓包重放 | 请求加入 nonce + 时间戳 |
| 修改 WebUI/action.sh | 核心授权逻辑不在脚本 |

### 2. 无法完全防御风险

| 风险 | 说明 |
|------|------|
| 逆向 patch .so | root 用户可反编译修改二进制 |
| 内核级 Hook | 可伪造 `/proc/cmdline` 读取结果 |
| 私钥泄露 | 私钥泄露后可伪造全部授权 |

### 3. 加固建议

- 公钥、服务器 URL、salt 做字符串混淆
- `authOk()` 多点调用
- companion 执行敏感操作前再次验签
- 发布版关闭日志
- license 和 clash_info 全字段签名
- 服务端请求限速
- 管理后台加管理员密钥或登录认证

## 七、实施顺序

建议按以下顺序开发：

1. 实现设备 ID 读取函数（C++）
2. 实现 license 验签逻辑（C++）
3. 实现 authOk() 并接入 Zygisk 拦截
4. 实现 EdgeOne 服务端 API
5. 实现卡密生成/删除管理后台
6. 实现 action.sh 参数模式
7. 实现 KernelSU WebUI
8. 实现互抵详情和锁死提示
9. 增加服务端 nonce/限速/日志
10. 打包测试

## 八、当前结论

下一阶段应优先做授权系统，而不是继续增加本地功能。

推荐技术路线：

```text
EdgeOne Edge Functions + KV
Ed25519 服务端签名
C++ .so 内置公钥验签
KernelSU WebUI 操作界面
互抵 + 渐进锁死防共享
```

该方案可以有效防止普通传播、共享卡密和简单篡改。对高级逆向无法做到绝对防御，但能显著提高破解成本。
